# Implementation Plan: Stride BepuPhysics Integration

**Branch**: `005-stride-bepu-integration` | **Date**: 2026-03-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/005-stride-bepu-integration/spec.md`

## Summary

Extend the physics sandbox to support the full range of BepuPhysics2 capabilities exposed through Stride.BepuPhysics. This includes 10 shape types (up from 3), 10 constraint types, per-body color and material properties, collision layer filtering, kinematic bodies, physics queries (raycast, sweep cast, overlap), and debug wireframe visualization. Changes span the proto contract, simulation, server, viewer, and all client interfaces (REPL, MCP, scripting).

**External Dependency**: BepuFSharp 0.2.0 (from BPEWrapper project, separate spec/feature). Must be completed and published to local NuGet feed before this feature's simulation/query work can begin. See `contracts/bepufsharp-extensions.md` for the API contract this feature expects. Spec provided at `~/projects/BPEWrapper/stride-bepu-integration-spec.md`.

## Phase 0 Research — Complete

See [research.md](research.md) for all findings. Key decisions:

1. **Debug visualization**: Custom wireframe via Stride's `WireFrameRenderObject` (Debug package's processor incompatible with gRPC-driven viewer)
2. **No new package references**: Stride.BepuPhysics already transitive via CommunityToolkit.Bepu
3. **BepuFSharp already has 10 constraint types**: Maps 1:1 to spec's curated set
4. **Material properties**: Use BepuPhysics2-native terminology (max_recovery_velocity, not "bounciness")
5. **All proto changes additive**: Full backward compatibility
6. **Query routing**: New internal mechanism needed (server → simulation request-response for queries)
