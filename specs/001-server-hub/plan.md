# Implementation Plan: Contracts and Server Hub

**Branch**: `001-server-hub` | **Date**: 2026-03-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-server-hub/spec.md`

## Summary

Establish the Physics Sandbox .NET solution with Aspire orchestration, shared gRPC contracts (proto files), a shared service defaults project, and the PhysicsServer hub — a stateless message router that accepts commands from clients, receives simulation state, and fans out updates to all subscribers. This is the foundation for all future services.

## Technical Context

**Language/Version**: F# on .NET 9.0 (services), C# on .NET 9.0 (AppHost, ServiceDefaults)
**Primary Dependencies**: .NET Aspire 9.x, Grpc.AspNetCore, Google.Protobuf, Grpc.Tools
**Storage**: N/A — server hub is a stateless in-memory message router
**Testing**: xUnit, Aspire DistributedApplicationTestingBuilder, Grpc.Net.Client
**Target Platform**: Linux (Arch Linux, rootless Podman for container resources)
**Project Type**: Distributed application (microservices, Aspire-orchestrated)
**Performance Goals**: <100ms state fanout latency, <1s command acknowledgment
**Constraints**: Single simulation source at a time, no command queueing for disconnected consumers
**Scale/Scope**: Developer sandbox — a handful of concurrent connections (1 simulation, 1 client, 1 viewer)

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Service Independence | PASS | PhysicsServer is an independent service. No shared mutable state. Communication via gRPC only. |
| II. Contract-First | PASS | Proto files in PhysicsSandbox.Shared.Contracts defined before implementation. gRPC for all real-time communication. |
| III. Shared Nothing | PASS | Only contracts project is shared. No cross-service project references. |
| IV. Spec-First | PASS | Feature spec complete with user stories and acceptance criteria. |
| V. Compiler-Enforced Contracts | PASS | PhysicsServer F# modules will have `.fsi` signature files. Surface-area baselines will be maintained. |
| VI. Test Evidence | PASS | Integration tests via Aspire testing builder. Contract tests for proto conformance. |
| VII. Observability | PASS | ServiceDefaults project provides OpenTelemetry, health checks, structured logging. |

**Engineering Constraint Exceptions**:

| Exception | Justification |
|-----------|---------------|
| AppHost in C# (not F#) | Microsoft provides no F# Aspire AppHost templates. `Projects.*` types are generated by MSBuild targets untested with F#. AppHost is ~30 lines of boilerplate — no domain logic. |
| ServiceDefaults in C# | Standard Aspire template project. Contains no domain logic — only extension method wiring. Shared by all services as infrastructure. |

**Structure Decision**: Follows the mono-repo layout from the design documents (MicroservicesFit.md). Each service is a top-level project under `src/`. AppHost and ServiceDefaults are standard Aspire infrastructure projects in C#. The server hub is F# with `.fsi` signature files per constitution. Integration tests use Aspire's testing builder.
